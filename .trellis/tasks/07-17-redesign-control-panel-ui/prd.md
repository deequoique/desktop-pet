# 重构桌宠控制面板 UI 与交互

## Goal

把 Electron 内置桌宠控制面板从功能平铺的长页面，重构为层级清晰、操作路径短、具有桌宠产品气质的控制界面。用户打开面板后应立即知道对方是否有桌宠设备在线、当前能否发送，以及下一步可以做什么。

## Background and Confirmed Facts

- 当前 React 控制面板在 `web/src/App.tsx` 中把成员/设备、连接、本机缩放、诊断、通话、表情、动作、个人音频、TTS 和位置依次平铺。
- `web/src/styles.css` 使用两列网格，但多个 section 依赖 `first-of-type`、`grid-row: span` 等隐式布局，新增功能后视觉顺序和代码顺序难以对应。
- 所有区域大量复用同一种深色渐变卡片、圆角按钮和 pill，缺少主次操作与场景分组；成员设备又以长文本行展示，扫描成本高。
- 当前标题文案“REMOTE CONSOLE / 黑白蓝主界面……”解释设计而非帮助用户操作，应改为面向用户的状态和动作。
- Electron 内置控制面板是产品入口；独立浏览器入口已废弃，只保留开发 fallback，不纳入设计或发布验收。
- 现有业务能力、Socket.IO/IPC 契约和设备目标规则应保持不变；此次重点是信息架构、交互流程、视觉系统与可用性。

## Initial Product Principles

- 日常操作面向对方成员，而不是要求发送方理解或选择对方的具体设备。
- 高频操作优先，低频设置和诊断收纳，不让首次用户面对所有技术选项。
- 通话是独立的沉浸场景；普通控制不应被大面积空视频框压缩。
- “发送给对方”的内容（动作、TTS、个人音频）和“管理本机/房间”的内容（成员、设备、服务器、缩放、诊断）必须分层。
- 顶层导航按任务拆分：“控制”只承载快捷动作与桌宠位置/表情控制；“发送”只承载文字语音和个人音频；“通话”承载呼叫与通话中画面；“设置”承载设备与技术配置。
- 视觉识别来自桌宠陪伴与设备间连接，而不是通用 SaaS dashboard、霓虹网格或装饰性技术术语。

## Requirements

- 重组控制面板导航与布局，消除当前长页面功能堆叠。
- 建立明确的页面/模式层级、主次按钮、状态、空状态、错误提示和危险操作表现。
- 为成员与设备提供可快速扫描的状态视图，离线设备历史不应压过当前在线设备。
- 日常发送目标始终是“对方”；只有对方存在在线桌宠设备时才允许发送，内容实际投递到该在线设备。
- 日常界面只在右上角保留一处对方在线状态，不重复展示“在线”“可接收”或设备解释文案。
- 对方只有一台设备在线时直接以该设备为目标，不显示设备选择器；两台及以上设备在线时，右上角在线状态成为按需展开的目标选择器，允许单选或多选在线设备。
- 多设备选择按对方成员记忆；新上线设备不自动加入。没有历史选择时默认单选最近上线的设备；已选设备全部离线且只剩一台在线设备时自动切换到该设备。
- 离线设备历史和完整设备明细进入设置或故障信息，不出现在日常目标选择器中。
- 表情、动作和位置集中在“控制”，不在“发送”中重复出现。
- TTS 与个人音频集中在“发送”，以“说句话 / 我的音频”二级切换组织，并保留录制、导入和试听能力。
- 通话开始前保持紧凑；通话中突出视频与媒体控制。
- 普通控制、TTS 和个人音频允许向已勾选的一台或多台在线桌宠设备投递；通话始终绑定一台同时具备桌宠端与控制端连接的目标设备。
- 当前勾选多台设备时，进入通话页先选择其中一台可通话设备，再发起通话。
- 服务器、身份、设备认领、桌宠大小和诊断应进入低频设置区域。
- 保留键盘焦点、响应式布局与 reduced-motion 支持。

## Constraints

- 不改变已完成的身份、在线状态、音频、TTS、WebRTC、缩放和诊断业务契约。
- 不为废弃的独立浏览器入口新增设计或产品能力。
- 不覆盖用户未提交的 `docs/releases.md` 与 `docs/troubleshooting.md` 改动。
- 不引入新的 UI 框架或全局状态库，继续使用现有 React、CSS 和 API adapter 边界。

## Acceptance Criteria

- [ ] 用户无需滚动或展开多个区域即可看见对方状态、桌宠是否可接收和最常用控制入口。
- [ ] 高频控制、内容发送、通话和设置形成明确层级，同一时刻不会平铺展示所有功能。
- [ ] 快捷动作只出现在“控制”；文字语音与个人音频只出现在“发送”，不存在跨页面重复入口。
- [ ] 日常界面不要求发送方选择对方设备；对方无设备在线时所有发送动作明确禁用，有设备在线时内容投递到在线设备。
- [ ] 对方仅一台设备在线时不出现多余选择步骤；两台及以上在线时可从右上角状态入口单选或多选发送目标。
- [ ] 首页只保留一处在线状态，不以说明文字、状态标签等形式重复表达同一信息。
- [ ] 在线设备和离线历史只在设置中按层级展示，不干扰面向“对方”的日常操作。
- [ ] 普通状态下通话区域紧凑，进入通话后视频与媒体控制成为主内容。
- [ ] 多目标发送逐设备执行并汇总成功、失败反馈；任一设备失败不掩盖其他设备的成功结果。
- [ ] 通话不会向多台设备同时发起；目标不具备完整通话端点时不能选择。
- [ ] TTS、个人音频和动作均能在两步以内发送到当前目标，并有一致反馈。
- [ ] 服务器、成员、设备恢复、缩放和诊断仍可访问，但不干扰日常控制。
- [ ] 视觉系统具有明确且一致的桌宠产品气质，不再依赖同质化卡片堆叠。
- [ ] Electron 实机完成主要流程、空状态、离线目标、窄窗口和键盘焦点验收。
- [ ] `npm run build:web`、`npm run build:pet` 与相关测试通过。

## Open Product Decisions

- 无。用户已确认原型视觉方向、四区导航、去除重复状态文案、多设备默认规则，以及“发送可多选、通话只单选”的边界。
