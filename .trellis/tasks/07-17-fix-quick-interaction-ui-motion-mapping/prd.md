# 修复快捷互动 UI 与动作映射

## 目标

让控制面板的“快捷互动”按钮具备明确、短暂的点击反馈，并且每个展示给用户的动作名称都与远端桌宠实际播放的动画语义一致。

## 已确认问题

- `web/src/App.tsx:1193` 按数组下标给第一个“开心”按钮永久添加 `primary`，因此它始终是黑底，并不代表选中或刚刚发送。
- `web/src/control-panel.css:6` 没有为 `.action-tile` 定义 hover/active/发送反馈，点击后只有 toast，按钮本身没有动画。
- 当前 sprite 桌宠实际公开的动作模板为：待机、左右移动、招手、跳跃、失败、等待；来源为 `pet/src/renderer/main.ts:128` 的 `SPRITE_MOTIONS`。
- `pet/src/renderer/main.ts:616` 将硬编码表情映射到 sprite 动作：开心→招手、吃惊→跳跃、委屈/生气→失败，而眨眼/平静没有对应 sprite 动画。这些映射与按钮文字不是一一对应。
- 过去引入传统 sprite 时曾决定把远程表情兼容映射到已有 sprite 状态；本次用户反馈表明该兼容映射不应继续作为当前 UI 的动作模板真相。

## 需求

- R1：所有快捷互动按钮默认使用一致的中性样式，不以固定位置制造永久选中态。
- R2：鼠标按下/点击发送时，按钮必须提供可感知但短暂的缩放、位移或高亮反馈；尊重 `prefers-reduced-motion`。
- R3：快捷互动展示项必须来自远端桌宠声明的实际能力，不能让 UI 文案与实际播放动画错配。
- R4：动作发送继续走现有 `animation` 命令、目标设备选择与 toast 反馈，不改变房间路由规则。
- R5：加载不到动作、没有目标桌宠或按钮禁用时，界面保持稳定且不制造虚假成功状态。

## 产品决策

- 快捷互动只展示远端桌宠真实声明、且适合用户主动触发的动画。
- 现有 `waving` 动画重命名为 `joy`，按钮名称为“开心”；现有 `failed` 动画重命名为 `sorrow`，按钮名称为“委屈”。
- “跳跃”和“等待”保持原动画 ID 与按钮名称。
- 最终快捷互动固定展示远端实际支持的“开心、跳跃、委屈、等待”；待机和左右移动仍作为内部状态保留，不显示为快捷互动。
- 移除当前硬编码的“吃惊、生气、眨眼、平静”等表情按钮，不新增缺失动画资源。

## 验收标准

- [x] AC1：首次进入快捷互动时没有任何按钮永久黑底；禁用状态仍清晰可辨。
- [x] AC2：点击可发送按钮时，按钮立即显示短暂按压/发送动画，结束后自动恢复。
- [x] AC3：每个可见动作按钮的名称与远端实际执行的 sprite 状态一致：“开心”发送 `animation:joy`，“跳跃”发送 `animation:jumping`，“委屈”发送 `animation:sorrow`，“等待”发送 `animation:waiting`。
- [x] AC4：远端未声明的快捷动作不显示；待机和左右移动不出现在快捷互动中。
- [x] AC5：`prefers-reduced-motion: reduce` 下取消位移动画，但保留非运动型状态反馈。
- [x] AC6：web 与 pet 构建通过；通过两个本地实例人工确认四个按钮触发对应动画。

## 范围外

- 本任务只重命名现有运行时 sprite 资源目录，不重新生成或补绘动画帧；Hatch QA 中遵循标准行命名的原始素材保持不动。
- 不恢复已经停用的 VRMA 动作 manifest 加载路径。
- 不改变 Socket.IO 的房间鉴权、目标设备路由或服务端持久化模型。
