# 改善首次引导与连接配置体验

## 目标

让启动、首次配置和连接管理路径清晰可理解：设置始终作为应用首页；用户不应在尚未建立服务器上下文时被迫选择身份；完成首次配置后，用户仍能通过明确入口修改本机身份。

## 背景与已确认事实

- 实际产品行为是：首次缺少有效服务器连接记录时，Electron 没有可靠地自动打开控制面板。源码虽然尝试在服务器地址或房间密钥为空时调用 `showControlWindow()`，但这不是完整的 pairing 就绪判定，也缺少验证真实显示行为的测试（`pet/src/main/index.js:793`、`pet/src/main/index.js:808`、`pet/test/pairing-main.test.cjs`）。
- 当前启动判断只检查服务器地址和房间密钥两个字符串，遗漏 `memberId` 等完整 pairing 必填字段；同时打包配置 `pet/config/production.json` 会提供服务器地址，因此“本机没有保存服务器记录”不等于 `configuredServerUrl()` 为空。非空但无效或残留的服务器/密钥也会绕过自动打开判断（`pet/src/main/index.js:126`、`pet/src/main/index.js:130`、`pet/src/main/index.js:134`、`pet/src/main/index.js:808`）。
- React 控制面板当前以“控制”页作为初始页面，而连接、身份和设备配置都位于“设置”，导致首次用户和需要维护配置的用户先落在不可操作页面（`web/src/App.tsx:230`）。
- 当前“设置 > 连接”把服务器地址、房间密钥、成员身份和设备名称放在同一张表单中；Electron pairing 配置尚未加载时，身份状态会先默认成成员 `a`（`web/src/App.tsx:224`、`web/src/App.tsx:228`、`web/src/App.tsx:1169`）。
- 连接中或已连接时，身份选择框被禁用；现有 pairing 配置只在点击“连接”时保存，因此没有明确的已连接身份修改流程（`web/src/App.tsx:778`、`web/src/App.tsx:1169`）。
- Electron 桌宠端和内置控制端共享同一份 pairing 配置。修改 `memberId` 时，两个端点都必须离开旧身份并使用新身份重新连接，不能只更新界面状态（`pet/src/main/index.js:717`、`pet/src/main/index.js:736`、`pet/src/renderer/main.ts:1129`）。
- 服务端加入房间时要求 `memberId` 必须为 `a | b`。历史产品决策明确保留手动选择成员身份；成员显示名称只是可变标签，不改变底层身份与路由（`.trellis/tasks/archive/2026-07/07-17-multi-device-presence-and-personal-audio/design.md`）。
- 当前服务端会拒绝运行中同一 `deviceId` 使用不同 `memberId` 加入，并返回 `device_identity_conflict`；因此身份修改需要服务端配合迁移设备归属，不能只修改本地 pairing 后重连（`server/src/index.js:552`、`server/src/index.js:555`）。

## 需求

- R1：设置必须是控制面板在所有启动场景下的首页；首次启动且没有完整的本地 pairing 配置时，Electron 还必须可靠地显示控制面板并直接进入连接配置流程。
- R2：在服务器上下文尚未就绪时，不把某个成员身份预选为用户的既定身份，也不要求用户先完成身份选择。
- R2.1：第一步只收集并验证服务器地址与房间密钥；验证失败时停留在第一步并显示可操作错误，不保存成完整 pairing 配置。
- R2.2：验证成功后显示服务端返回的两名成员名称，要求用户明确选择其中一个身份；不得静默默认选择成员 `a`。
- R3：提供清晰、可发现的本机身份修改入口。
- R4：身份修改必须作为一次完整 pairing 变更应用到 Electron 桌宠端和控制端，并在新身份下重新连接。
- R4.1：设置页在已连接状态下提供独立的“更改身份”操作和确认说明；连接表单不依赖用户先手动断开才能编辑身份。
- R4.2：服务端必须以原子方式处理稳定设备 ID 的成员归属变更，避免新身份加入被 `device_identity_conflict` 拒绝或在两个成员下留下重复设备记录。
- R4.3：迁移只移动当前设备记录，不移动旧成员的私有音频、显示名称或其他设备。
- R4.4：服务端迁移失败时不得写入本地新身份；服务端迁移成功但 Electron 写盘失败时必须尝试迁回旧成员，并向用户明确报告结果，不得伪报切换成功。
- R5：正常运行中的临时断线保留当前页面，只呈现连接状态或可操作错误，不自动跳转设置。
- R6：已有完整有效 pairing 配置的用户继续进入正常使用路径，不要求重新配置或迁移。
- R7：保留现有手动选择成员 `a | b` 的身份模型，以及成员显示名称不影响底层身份与路由的约束。
- R8：保留工作区内与本任务无关的未提交修改。

## 验收标准

- [ ] AC1：任意启动场景下控制面板首屏都是设置；全新用户在没有完整 pairing 配置时会自动看到连接配置流程，且不会先被默认认定为用户 A。
- [ ] AC2：用户完成服务器相关配置后，可以明确选择自己的成员身份并完成连接。
- [ ] AC2.1：服务器地址或密钥无效、服务器不可达时，用户不会进入身份选择步骤，也不会留下看似完整但无法使用的 pairing 配置。
- [ ] AC2.2：身份选择项使用验证服务器返回的实际成员显示名称；服务端尚无自定义名称时才显示“用户 A / 用户 B”。
- [ ] AC3：已有身份的用户可以从设置中发起身份修改，并明确知道该操作会重新连接。
- [ ] AC4：身份修改成功后，桌宠端与控制端使用同一新 `memberId + deviceId` 重新加入；不会出现两个端点身份不一致或仍残留旧连接。
- [ ] AC4.1：用户无需先手动点击“断开”；确认身份修改后由应用完成断开、服务端设备归属变更、本地保存和重连。
- [ ] AC4.2：服务端拒绝不合法或不属于当前设备的身份迁移，迁移过程失败时不会产生跨成员重复设备记录。
- [ ] AC4.3：迁移后新成员下保留相同 `deviceId`、设备名称和首次使用时间，旧成员下不再保留该设备；旧成员私有音频仍留在旧成员库中。
- [ ] AC4.4：服务端拒绝或超时时，本地 pairing 与当前连接保持旧身份；Electron 写盘失败且补偿成功时，服务端和本地均恢复旧身份。
- [ ] AC5：已有完整有效配置的用户启动后保持自动连接，并同样以设置作为首页。
- [ ] AC6：正常使用中发生临时断线时，当前页面保持不变，不自动跳到设置。
- [ ] AC7：自动化测试至少覆盖完整 pairing 就绪判定、首次无配置时显示控制面板并进入设置、首次身份选择和身份修改重连；测试不能只断言源码中存在某个函数调用。

## 范围外

- 不改变房间固定两名成员和 `memberId: 'a' | 'b'` 的服务端身份模型。
- 不把成员显示名称升级为权限或路由标识。
- 不因本任务恢复已经废弃的独立浏览器入口；发布与验收目标仍是 Electron 内置控制面板。
- 不处理运行中每次断线都强制跳转设置的行为。
