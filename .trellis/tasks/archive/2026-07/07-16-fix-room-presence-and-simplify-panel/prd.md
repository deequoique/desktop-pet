# 改造多设备房间并增强桌宠诊断

## Goal

让单 server、固定双用户的产品模型正确支持每人多台设备，区分“用户在线”与“设备端点在线”，提供明确的单设备控制目标和成员私有音频库；同时为 Win10 桌宠异常变大/无法缩放提供可靠控制入口与可导出的脱敏诊断证据。

## Background and Evidence

- `server/src/index.js:145-159` 把对方 pet 或 controller 任一 socket 存在定义为 `peerOnline`，而 `web/src/App.tsx:842-843` 直接用它显示“对方在线”；因此后台 pet 在线会被误解为对方本人在线。
- `server/src/index.js:801-820` 已在 socket 断开时清除端点并立即广播，根因不是漏掉 disconnect，而是当前“一台安装实例等于一个 participant”的模型不符合“两名用户各有多台设备”的产品约束。
- `pet/src/main/index.js:121-140` 为每个安装生成独立 `participantId`；共享房间密钥不能自动判断设备属于哪名用户。用户确认采用手动选择用户 A/B，不增加账户或个人配对码。
- `web/src/App.tsx:869` 的单一“桌宠端” pill 无法表达多设备；用户需要看到连接过该 server 的设备、端点在线状态和最后在线时间。
- `web/src/App.tsx:978-998` 的“预录台词”只是远程列出目标 pet 的本地文件。用户决定将其补全为“我的音频”，按成员隔离并跨同一成员设备共享。
- `pet/src/main/index.js:503-516` 是唯一主进程窗口缩放入口，`pet/src/renderer/main.ts:1195-1238` 的 +/- 控件依赖透明窗口命中。现有代码没有定时自动放大路径；单台 Win10 故障缺少日志，不能在无证据时宣称具体 DPI 根因。

## Task Map

1. `07-17-multi-device-presence-and-personal-audio`
   - 负责成员/设备/端点身份、持久设备状态、单目标路由、设备对通话和成员私有音频库。
2. `07-17-pet-scaling-and-diagnostics`
   - 负责控制面板缩放、恢复默认大小、结构化日志和用户主动导出诊断文件。

两个子任务可以独立实现；父任务最终负责同时升级后的构建、交叉 UI 检查、文档一致性与完整验收。

## Requirements

### 双用户多设备

- 房间身份拆为固定成员 A/B、稳定设备 ID 和 pet/controller 端点；每名成员可连接多台设备。
- 客户端手动选择成员身份，设备名称默认来自主机名且可编辑；现有 `participantId` 原值迁移为 `deviceId`。
- 固定 `memberId` a/b 与显示名称分离；任意一方的已连接控制端都可修改双方名称，服务端持久保存并同步，最后一次成功写入生效。
- 设备 ID 使用持久随机 UUID，不依赖 IP、MAC 或硬件指纹；换网络不改变身份。本地数据丢失后可认领同成员离线设备，在线设备不可认领。
- 服务端持久登记设备名称、成员归属和最后在线时间；连续离线 30 天自动清理。
- 顶部“对方在线”只由对方 controller 在线推导；设备列表分别展示 pet/controller 状态。
- 控制、动作查询、TTS、个人音频和双向通话只作用于明确选择的对方设备；单设备自动选择，多设备保留选择，目标离线不静默切换或广播。
- 新协议要求 `memberId + deviceId`，不向后兼容；旧客户端得到 `upgrade_required`，所有客户端与 server 协调升级。

### 成员私有音频

- “预录台词”改为“我的音频”，支持麦克风录制、导入、试听、重命名、删除和发送到所选目标 pet 播放。
- 服务端按成员持久保存音频与 metadata；同成员跨设备共享，正常 UI/API 不列出另一成员的库或文件名。
- 单段最长 60 秒、最大 10 MB、每成员最多 100 段；支持 MP3、WAV、OGG、M4A、WebM，客户端与服务端分别校验可验证边界。
- 成员手动选择下的隔离是可信双人部署的产品逻辑，不承诺抵抗持有共享密钥者恶意冒充，也不构成 DRM。

### 缩放与诊断

- 控制面板提供当前尺寸、可靠缩放和恢复默认大小；桌宠浮层入口与控制面板复用同一主进程真相源。
- 主进程记录应用/OS/Electron 版本、显示器 bounds/workArea/scaleFactor、窗口 bounds、保存 scale、缩放请求/结果、显示指标变化和受控错误摘要。
- 托盘和控制面板均可由用户主动导出诊断文件；room secret、API key、凭据和音频内容必须脱敏或排除，不自动上传遥测。
- 在取得故障机日志前只交付证据支持的边界修正、可靠控制入口和可观测性，不猜测式加入 DPI 比例补偿。

## Acceptance Criteria

- [x] 用户 A/B 各自可连接多台设备；同一设备 pet/controller 合并展示，旧协议明确被要求升级。
- [x] 任意一方可修改双方显示名称，所有在线设备与 server 重启后的名称一致，且改名不影响成员身份或路由。
- [x] 仅后台 pet 在线不会显示“对方在线”；任一对方 controller 在线会显示在线，断连后立即更新。
- [x] server 重启后保留 30 天内设备/最后在线状态；换 IP 不新增设备，清除本地数据后可认领离线设备。
- [x] 多设备时所有控制、语音和通话只命中所选设备，不自控、不广播、不串设备或房间。
- [x] 用户可录制或导入符合限制的音频并管理、试听、发送；同成员跨设备共享，另一成员不能通过正常 UI/API 列出。
- [x] 音频超出格式、大小、时长或数量限制时被明确拒绝，不破坏房间、控制或通话。
- [x] 控制面板和桌宠浮层均能调整同一实际 scale；恢复默认后窗口在可见工作区，重启保持一致。
- [x] 用户能从托盘或控制面板导出包含显示器/窗口缩放链路的诊断文件，自动化检查确认敏感字段和音频未泄露。
- [x] 未获得 Win10 故障证据前不虚报已修复未知根因；导出日志足以区分 IPC、保存状态、实际 bounds 与 DPI/display 变化。
- [x] `npm test --prefix server`、`npm run build:web`、`npm run build:pet` 通过，并完成两子任务交叉手工验收。

验收环境为 macOS Electron 实机加本机多设备 Socket.IO 端点；Win10 安装、DPI 与多显示器行为仍需在 Windows 故障机或虚拟机复验，不作为未知根因已修复的证据。

## Out of Scope

- 多租户房间目录、账户登录、强成员鉴权、防止共享密钥持有人冒充 A/B。
- 多设备广播控制、多人 Mesh 通话或让应用 server 中继 WebRTC 媒体。
- 使用硬件指纹自动恢复被彻底清除的设备身份。
- 自动上传遥测，或在没有故障日志/复现步骤时声称修复特定 Win10 DPI 根因。
