# 支持双用户多设备与个人音频库

## Goal

将房间身份拆分为成员与设备，增加设备状态、单目标路由和成员私有音频库。

## Requirements

- 一台 server 只服务固定两名成员；共享密钥保留，客户端手动选择用户 A/B，不引入账户或个人密钥。
- `memberId` 固定为 `a` / `b`，但两个成员的显示名称是房间级共享配置；任意已加入的 controller 都可修改任一成员名称。
- 成员显示名称由服务端持久保存并实时同步到双方所有设备；并发修改采用最后一次成功写入，改名不改变身份、设备归属、音频库或路由。
- 身份模型为成员、设备、端点三层：每名成员可有多台稳定 `deviceId`，每台设备分别维护 pet/controller socket。
- 现有 `participantId` 迁移为 `deviceId`；首次升级必须选择成员身份并确认可编辑设备名称。
- 设备登记由服务端持久保存，展示在线端点和最后在线时间，离线 30 天后自动清理。
- 本地身份丢失时可认领同成员的离线设备；在线设备不可被认领。
- 顶部“对方在线”只表示对方至少有一个 controller 在线；设备列表独立展示 pet/controller 状态。
- 控制、语音和通话只路由到明确选择的一台对方设备；单设备自动选择，多设备记住选择，目标离线后禁止误切换或广播。
- 双向通话绑定发起设备与目标设备，只通知这两个设备的相关端点。
- “预录台词”升级为成员私有“我的音频”：支持录制、导入、试听、重命名、删除，并发送到所选目标设备播放。
- 音频由服务端按成员持久保存，同成员跨设备共享；正常 UI/API 不返回另一成员的音频列表。
- 单段最长 60 秒、最大 10 MB、每成员最多 100 段；支持 MP3/WAV/OGG/M4A/WebM。
- 新协议不向后兼容；旧客户端返回 `upgrade_required`，服务端和所有客户端协调升级。

## Constraints

- 手动成员选择和音频隔离是可信双人部署下的产品逻辑，不承诺抵抗持有共享密钥者恶意冒充 A/B。
- 设备身份不使用 IP、MAC 地址或硬件指纹；换 IP 不影响身份，彻底清除本地数据后通过手动认领恢复。
- 不把 WebRTC 媒体改为经过应用服务端；音频库短音频的存储/转发不改变该约束。

## Acceptance Criteria

- [ ] 用户 A/B 各自可连接多台设备；非法成员身份被拒绝，旧协议得到 `upgrade_required`。
- [ ] 任意一方可修改双方显示名称，重启 server 后名称仍保留且所有在线设备一致；改名不会生成新成员或改变路由。
- [ ] 同一设备的 pet/controller 合并展示；断连立即更新，服务端重启后仍保留 30 天内的离线设备和最后在线时间。
- [ ] 仅后台 pet 在线不会让顶部显示“对方在线”；任一对方 controller 在线会正确显示。
- [ ] 多台设备时可选择唯一目标，所有命令和双向通话只涉及发起设备与目标设备。
- [ ] 现有安装 ID 可迁移；换网络不新增设备；本地数据丢失后可认领离线设备但不能覆盖在线设备。
- [ ] 用户可录制或导入符合限制的音频，并试听、重命名、删除和发送播放。
- [ ] 同一成员多设备看到相同音频库，另一成员无法通过正常 UI/API 列出该库。
- [ ] 音频格式、时长、大小和数量超限时返回明确错误，不影响房间、控制或通话。
- [ ] server 测试、web 构建和 pet 构建通过。

## Out of Scope

- 多租户房间目录、账户登录、强成员鉴权和恶意用户隔离。
- 多设备广播控制或多人 Mesh 通话。
- 用应用 server 中继 WebRTC 屏幕/麦克风媒体。
