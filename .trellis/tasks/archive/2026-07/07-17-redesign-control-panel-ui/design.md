# 技术设计：重构桌宠控制面板 UI 与交互

## 1. 范围与边界

本任务重构 Electron 内置的 React 控制面板，主要落点为 `web/src/App.tsx`、`web/src/styles.css` 与 `web/src/api.ts`。Electron main/preload、桌宠 renderer 和 server 的现有身份、房间、命令、个人音频、TTS 与 WebRTC 协议保持兼容。

独立浏览器入口已经废弃，不作为产品界面或验收对象。`web` 仍可保留开发 fallback，但不为它增加独立设计分支。

本任务不引入路由库、UI 框架或全局状态库。页面导航、弹层和通话阶段继续由 React 本地状态驱动；视觉系统继续使用普通 CSS。

## 2. 信息架构

控制面板使用固定左侧导航与共享顶部栏：

```text
共享框架
  顶部：双方名称 / 对方在线状态与多设备目标入口
  控制：快捷表情、动作、位置、紧凑文字/音频入口、本机缩放、最近互动
  发送：说句话 / 我的音频
  通话：通话前选择与呼叫 / 通话中远端画面与媒体控制
  设置：身份名称、连接密钥、设备、桌宠缩放、诊断
```

- “控制”是默认页，只显示高频桌宠互动，不重复完整 TTS、音频库或通话画面。
- “发送”只承载文字语音与当前成员私有音频；动作不在此重复。
- “通话”在 idle 状态保持紧凑；`requesting-media | calling | in-call` 时切换为大画面布局。
- “设置”收纳低频和技术性内容；服务器、密钥、设备历史、认领、成员改名、TTS BYOK、缩放复位和诊断不占据日常页面。

## 3. 组件与状态边界

`App` 保留连接、房间、媒体和业务副作用，视图拆成同目录内的小组件；若单文件仍过长，再移动到 `web/src/components/`，不提前建立无必要抽象。

建议组件边界：

- `AppShell`：左侧导航、顶部身份和页面切换。
- `PeerTargetMenu`：唯一的对方在线状态与多设备选择入口。
- `ControlView`：快捷动作、完整动作抽屉、位置、本机缩放和最近反馈。
- `SendView`：TTS composer、声音选择、个人音频列表、录制/导入及条目操作。
- `CallView`：目标选择、呼叫、视频 stage、媒体开关与结束通话。
- `SettingsView`：连接、成员名称、设备列表/认领、缩放、BYOK 与诊断。
- `Toast` / `InlineStatus`：统一成功、部分成功、错误和忙碌反馈。

导航状态为 `'control' | 'send' | 'call' | 'settings'`。不写入 URL；可在当前窗口生命周期内记忆，初始固定为 `control`。开始通话或收到通话开始事件时强制进入 `call`；通话中切换其他页时媒体不断开，但左侧通话项保留进行中标记。

## 4. 在线状态与设备目标

可发送设备定义为对方成员下 `petOnline === true` 的设备。顶部只出现一处状态：

- 0 台：`小伊离线`，所有远程发送入口禁用。
- 1 台：`小伊在线`，自动选中，不出现设备数量或选择步骤。
- 2 台及以上：`小伊在线 · N 台`，点击展开仅包含当前在线设备的多选菜单。

选择以对方 member ID 为 key 存入 localStorage，值为设备 ID 数组。归一化规则：

1. 保留仍在线的历史选择。
2. 新上线设备不自动加入。
3. 没有有效历史选择时，单选 `lastSeenAt` 最新的在线设备。
4. 已选设备全部离线且只剩一台在线时，自动选中该设备。
5. 离线历史只在设置页显示，不进入目标菜单。

页面不再显示“对方在线”“桌宠可接收”“无需选择设备”等重复说明。离线时通过唯一状态、禁用态和必要的操作反馈表达原因。

## 5. 多目标发送与协议兼容

server 当前所有命令都以单个 `targetDeviceId` 路由。为避免扩大服务端协议，多选在 `web/src/api.ts` 的客户端 adapter 中逐设备 fan-out：

- `pet:command`：对每个选中 ID 发送一次现有事件。
- `tts:create`：每台设备创建独立 job，`Promise.allSettled` 汇总结果。
- `audio:play`：每台设备创建独立播放 job并汇总结果。
- 动作目录查询使用当前选择中的主设备；主设备为用户最近明确点击的设备，否则为选择数组首项。当前产品假设各设备使用相同桌宠模型；某设备无法执行命令时只报告该设备失败，不阻断其他设备。

adapter 不再用一个隐式全局 `targetDeviceId` 同时服务所有能力。普通发送 API 显式接收 `targetDeviceIds: string[]`；WebRTC 与 `requestCall` 显式接收单个 `targetDeviceId`。这在类型层阻止把多目标传给通话。

多目标反馈规则：

- 全部成功：显示一次“已发送到 N 台设备”。
- 部分成功：显示“已发送 X 台，Y 台失败”，失败设备可从详情查看。
- 全部失败：显示统一错误，并保留输入内容以便重试。
- 无目标：不调用 adapter，按钮保持禁用。

## 6. 通话目标与阶段

可通话设备必须同时满足 `petOnline && controllerOnline`。一次房间通话仍绑定现有的一对设备：

- 只有一台可通话设备时直接显示“开始通话”。
- 多台可通话设备时，通话页要求单选一台；可优先带入普通发送选择中的主设备，但不继承多选。
- 当前没有完整端点的设备不作为可选目标；不向多台设备同时发起 `call:start`。

idle 页显示对方、可选设备和主呼叫按钮。接通后远端屏幕占主要面积，底部只保留本机麦克风、对方系统声音、对方麦克风、全屏和结束。ICE 类型、轨道数量和详细故障诊断移入设置/诊断；通话页只显示连接中、稳定、仅音频或失败等用户可行动状态。

TURN 音频兜底、屏幕权限失败、远端停止共享和恢复超时等现有状态机不变，只改变呈现位置和文案密度。

## 7. 发送与设置细节

发送页：

- “说句话”包含 textarea、声音选择/试听、字符数和发送状态。
- “我的音频”只读取当前成员私有库，提供试听、发送、录制、导入、重命名和删除。
- BYOK API Key 配置属于设置页；发送页只显示已可用声音和必要的配置缺失入口。

设置页按低频任务分组：

- 双方显示名称，任一方仍可修改双方名称。
- 当前服务器和房间密钥、连接/断开。
- 本机身份、设备名、同成员离线设备认领。
- 双方设备状态与最近连接时间。
- 本机桌宠缩放、恢复默认和导出诊断日志。
- TTS provider/BYOK 配置。

危险或不可逆操作（删除音频、替换凭据、设备认领）使用确认层；普通切换与发送不使用浏览器 `prompt`/`confirm`，改为界面内表单或弹层。

## 8. 视觉、响应式与可访问性

采用用户已确认的原型方向：暖灰纸张背景、深石墨导航、梅紫主色、桃色点缀和柔和桌宠形象。通过留白、字号和背景层级区分任务，不再让所有内容使用同权重渐变卡片。

- 最小目标窗口按现有 Electron 控制窗口实测；宽屏为左导航 + 主内容，窄窗将右侧辅助栏折叠到主内容下方。
- 所有图标按钮有可见文本或 `aria-label`；目标菜单、二级 tab、弹层和通话控件可键盘操作。
- 焦点样式不依赖颜色变化；在线/错误状态同时使用文字或图形。
- 动画只用于菜单、toast 和页面过渡，并尊重 `prefers-reduced-motion`。
- 删除纯装饰性技术文案；同一职责只保留一个状态来源。

## 9. 兼容、风险与回滚

- 不改变 server 单目标协议，旧桌宠 renderer 无需升级协议。
- 多目标 TTS/音频会按设备产生多个 job；客户端限制并发并汇总，避免一次点击重复触发。若容量限制导致部分失败，按部分成功呈现。
- `App.tsx` 当前集中了 WebRTC 副作用，重构时先移动纯视图，再移动状态边界，避免同时改状态机和布局。
- 保留旧 CSS 到各页面完成后再删除，便于逐页回滚；任一步出现媒体回归时可恢复旧视图而不回滚 server。
- 不触碰用户未提交的 `docs/releases.md` 和 `docs/troubleshooting.md`。

## 10. 原型基准

已确认三类视觉原型：默认控制页、发送页、通话中页面。实现以其信息层级和交互职责为准，不要求逐像素复刻；真实数据、错误态、窄窗口和键盘交互优先于静态稿。
